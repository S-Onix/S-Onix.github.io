## 📌 소개

- 소개 : 클로드 문답형식을 활용한 스프링 공부
- 목표 : MSA 개념 학습

## 📝 본문



## 1. Monolithic Architecture

모든 기능이 하나의 애플리케이션에 들어있는 구조입니다.

**장점**: 개발/배포가 간단, @Transactional로 트랜잭션 관리, 디버깅 쉬움

**한계** (서비스가 커지면):

- 배포 공포증: 하나를 수정하면 전체에 영향
- 빌드/배포 속도 저하
- 기술 스택 고정 (전부 같은 언어)
- 확장의 비효율 (특정 서비스만 스케일링 불가)
- 장애 전파 (한 모듈 문제가 전체 다운으로)

------



## 2. Microservice Architecture란?

애플리케이션을 **독립적으로 배포 가능한 작은 서비스들의 집합**으로 분리하는 아키텍처 패턴입니다. 각 서비스는 자기만의 코드, 자기만의 DB, 자기만의 배포 파이프라인을 가집니다.

```
┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
│ 주문  │  │ 결제  │  │ 회원  │  │ 알림  │  ← 각각 독립된 JAR
└──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘
   ↓         ↓         ↓         ↓
 [주문DB]  [결제DB]  [회원DB]  [알림DB]     ← 각자 DB를 가짐
```

------

## 3. 핵심 원칙

- **독립 배포**: 각 서비스를 다른 서비스에 영향 없이 배포
- **서비스별 독립 DB**: 다른 서비스의 DB에 직접 접근 금지
- **API 통신**: 서비스 간 HTTP API 또는 메시지 큐로 통신
- **자율적 기술 선택**: 서비스별 최적 기술 스택 선택 가능

------



## 4. 서비스 간 통신

### 동기 통신 (REST API / gRPC)

호출하고 응답을 기다립니다. 결제 처리, 재고 확인처럼 **결과가 있어야 다음 단계로 넘어가는 경우**에 사용합니다.

장점: 구현 간단, 결과 즉시 확인 단점: 호출 대상이 죽으면 함께 실패, 서비스 간 결합도 높음

### 비동기 통신 (Kafka / RabbitMQ)

메시지 큐에 이벤트를 발행하고 응답을 기다리지 않습니다. 알림 발송, 로그 수집, 통계 집계처럼 **결과를 기다릴 필요 없는 경우**에 사용합니다.

장점: 서비스 간 결합도 낮음, 한 서비스가 죽어도 영향 적음 단점: 구현 복잡, 메시지 유실/중복 처리 필요, 디버깅 어려움

------



## 5. MSA의 핵심 과제들

### 분산 트랜잭션 → Saga 패턴

Monolith에서는 @Transactional 하나로 해결되지만, MSA에서는 각 서비스가 독립된 DB를 가지므로 하나의 트랜잭션으로 묶을 수 없습니다. Saga 패턴은 각 서비스의 로컬 트랜잭션을 순차 실행하고, 실패 시 보상 트랜잭션(Compensating Transaction)을 역순으로 실행하여 되돌립니다.

```
정상: 주문 생성 → 결제 처리 → 재고 차감
실패: 재고 차감 실패! → 결제 취소(보상) → 주문 취소(보상)
```

### 서비스 디스커버리

서비스가 수십 개일 때 서로의 위치를 파악하기 위해 Service Registry(Eureka, Consul)를 사용합니다.

### API Gateway

클라이언트에게 단일 진입점을 제공합니다. 라우팅, 인증, Rate Limiting 등을 담당하며, Reverse Proxy와 유사한 역할입니다.

### Circuit Breaker

외부 서비스 호출 시 실패가 일정 횟수 이상 되면 호출 자체를 차단하는 패턴입니다. 장애 전파(Cascading Failure)를 방지합니다. CLOSED(정상) → OPEN(차단) → HALF-OPEN(복구 테스트) 세 가지 상태로 동작합니다.

### 분산 로깅 & 모니터링

모든 서비스를 관통하는 Trace ID를 부여하여 요청 흐름을 추적합니다. (Zipkin, Jaeger, Micrometer Tracing)

------

------



## 7. Monolith vs MSA

|               | Monolith              | Microservice             |
| ------------- | --------------------- | ------------------------ |
| **배포**      | 전체를 한번에         | 서비스별 독립 배포       |
| **확장**      | 전체를 스케일링       | 필요한 서비스만 스케일링 |
| **기술 선택** | 하나로 통일           | 서비스별 자유 선택       |
| **트랜잭션**  | @Transactional 하나로 | Saga 패턴 (복잡)         |
| **디버깅**    | 쉬움                  | 분산 추적 필요 (어려움)  |
| **팀 규모**   | 소규모 적합           | 대규모 적합              |

**Monolith 적합**: 프로젝트 초기, 소규모 팀, 도메인 불명확, 빠른 개발 **MSA 적합**: 대규모 팀, 독립 배포/확장 필요, 트래픽 집중

**현실적 조언**: "Monolith First" — 처음에는 Monolith로 시작하여 도메인 경계를 파악한 뒤, 필요한 부분부터 점진적으로 MSA로 분리합니다.

## 🎯 면접 예상 질문

### Q1. Microservice Architecture란 무엇이며, Monolith와의 차이는 무엇인가요?

**모범 답안:**

> Microservice Architecture는 애플리케이션을 독립적으로 배포 가능한 작은 서비스들의 집합으로 분리하는 아키텍처 패턴입니다. Monolith는 모든 기능이 하나의 JAR에 들어있어 함께 빌드/배포되지만, MSA에서는 각 서비스가 자기만의 코드, 자기만의 DB, 자기만의 배포 파이프라인을 가집니다. 서비스 간 통신은 HTTP API나 메시지 큐로 이루어집니다.

**암기:** "독립 배포 + 독립 DB + API 통신"

### Q2. 서비스 간 동기 통신과 비동기 통신의 차이를 설명하고, 각각 언제 사용하나요?

**모범 답안:**

> 동기 통신은 REST API나 gRPC로 호출하고 응답을 기다리는 방식입니다. 결제 처리, 재고 확인처럼 결과가 있어야 다음 단계로 넘어갈 수 있는 경우에 사용합니다. 비동기 통신은 Kafka나 RabbitMQ 같은 메시지 큐에 이벤트를 발행하고, 응답을 기다리지 않는 방식입니다. 알림 발송, 로그 수집처럼 결과를 기다릴 필요 없는 경우에 사용합니다. 비동기 방식은 서비스 간 결합도가 낮아 한 서비스가 죽어도 다른 서비스에 영향이 적습니다.

### Q3. MSA에서 분산 트랜잭션 문제란 무엇이며, 어떻게 해결하나요?

**모범 답안:**

> Monolith에서는 @Transactional 하나로 여러 테이블의 트랜잭션을 관리할 수 있었지만, MSA에서는 각 서비스가 독립된 DB를 가지므로 하나의 트랜잭션으로 묶을 수 없습니다. 이를 해결하기 위해 Saga 패턴을 사용합니다. 각 서비스의 로컬 트랜잭션을 순차적으로 실행하고, 중간에 실패하면 이미 완료된 서비스에 보상 트랜잭션(Compensating Transaction)을 역순으로 실행하여 되돌립니다.

**암기:** "독립 DB → @Transactional 불가 → Saga → 보상 트랜잭션 역순 롤백"

### Q4. Circuit Breaker 패턴이란 무엇이며, 왜 필요한가요?

**모범 답안:**

> Circuit Breaker는 외부 서비스 호출 시 실패가 일정 횟수 이상 되면 호출 자체를 차단하는 패턴입니다. 이것이 필요한 이유는 장애 전파(Cascading Failure) 때문입니다. 결제 서비스가 응답이 느려지면 주문 서비스의 스레드가 전부 응답을 기다리며 점유되어, 결국 주문 서비스까지 다운됩니다. Circuit Breaker는 CLOSED(정상 호출), OPEN(호출 차단), HALF-OPEN(복구 테스트) 세 가지 상태로 동작합니다.

**암기:** "장애 전파 방지 + CLOSED → OPEN → HALF-OPEN"

### Q5. Monolith와 MSA 중 어떤 상황에서 어떤 것을 선택해야 하나요?

**모범 답안:**

> Monolith는 프로젝트 초기 MVP 단계, 팀 규모가 작은 경우, 도메인 경계가 아직 명확하지 않은 경우에 적합합니다. MSA는 서비스 규모가 크고 팀이 여러 개인 경우, 서비스별 독립 배포/확장이 필요한 경우, 특정 서비스에 트래픽이 집중되는 경우에 적합합니다. 실무에서는 "Monolith First" 전략으로 처음에는 Monolith로 시작하여 도메인 경계를 파악한 뒤, 필요한 부분부터 점진적으로 MSA로 분리하는 것이 권장됩니다.

**암기:**

- Monolith → "초기, 소규모, 도메인 불명확, 빠른 개발"
- MSA → "대규모 팀, 독립 배포, 독립 확장, 트래픽 집중"
