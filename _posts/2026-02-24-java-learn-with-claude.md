#  javac, Bytecode, JIT Compilation



##  Java 컴파일 과정 전체 흐름

Java는 두 단계의 컴파일 과정을 거친다. 이는 **"Write Once, Run Anywhere (WORA)"** 를 실현하기 위한 설계다.

```
.java ──[javac]──▶ .class (Bytecode)
  │
  └──[ClassLoader]──▶ JVM 메모리에 적재
      │
      ├──[Interpreter]──▶ 바이트코드 한 줄씩 해석/실행
      │
      └──[JIT Compiler]──▶ Hot Code를 기계어로 변환
           ├── C1: 빠른 컴파일, 가벼운 최적화
           └── C2: 느린 컴파일, 강력한 최적화
```

C/C++처럼 한 번에 기계어로 컴파일하면 빠르지만 플랫폼마다 다시 컴파일해야 한다. Java는 바이트코드라는 중간 표현(IR: Intermediate Representation)을 두어 이식성을 확보하고, 런타임 비용은 JIT로 줄인다.



##  javac — 정적 컴파일러

### 역할

javac은 Java 소스코드(.java)를 바이트코드(.class)로 변환하는 정적 컴파일러다. 문법 검사, 타입 검사, 바이트코드 생성을 수행한다.

### javac의 최적화가 보수적인 이유

javac은 **공격적으로 최적화하지 않고**, 대부분의 최적화를 JIT에 맡긴다. 이유는 다음과 같다:

**1. 정적 정보만 보유**

javac은 소스코드의 문법과 타입 정보만 알고 있으며, 실제로 어떤 코드가 얼마나 자주 실행되는지, 어떤 객체가 생성되는지 전혀 모른다.

**2. 다형성 미해결**

```java
public void process(Animal animal) {
    animal.speak();  // Dog? Cat? 컴파일 타임에는 알 수 없음
}
```

javac은 `animal`이 `Dog`인지 `Cat`인지 모르지만, JIT는 런타임에 "99%는 Dog가 들어온다"는 걸 알 수 있다.

**3. JIT 최적화 여지 보존**

javac이 공격적으로 최적화하면 바이트코드가 원본과 달라져서, JIT가 오히려 최적화하기 어려워질 수 있다.

### javac이 아는 것 vs JIT가 아는 것

| 구분                | javac (정적 정보) | JIT (런타임 정보) |
| ------------------- | ----------------- | ----------------- |
| 소스코드 문법/타입  | ✅                 | ✅                 |
| 메서드 호출 빈도    | ❌                 | ✅                 |
| 분기 통계 (if/else) | ❌                 | ✅                 |
| 다형성 실제 구현체  | ❌                 | ✅                 |
| 객체 탈출 여부      | ❌                 | ✅                 |



##  .class 파일 구조

javac이 생성하는 `.class` 파일에는 바이트코드만 들어있지 않다.

```
.class 파일 구조
┌─────────────────────────┐
│ Magic Number (CAFEBABE)  │  ← Java 클래스 파일 식별 (4바이트)
│ Version (minor + major)  │  ← Java 버전 (Java 17 = Major 61)
│ Constant Pool            │  ← 가장 큰 영역
│ Access Flags             │  ← public, abstract, final 등
│ This Class / Super Class │  ← 현재 클래스, 부모 클래스
│ Interfaces               │  ← 구현 인터페이스
│ Fields                   │  ← 필드 이름, 타입, 접근 제어자
│ Methods (+ 바이트코드)    │  ← 메서드 정보 + 바이트코드 자체
│ Attributes               │  ← 소스 파일명, 디버깅 정보, 어노테이션
└─────────────────────────┘
```

| 구성 요소         | 설명                                                         |
| ----------------- | ------------------------------------------------------------ |
| **Magic Number**  | `0xCAFEBABE` — Java 클래스 파일임을 식별하는 4바이트 시그니처 |
| **Version**       | Minor + Major 버전 (Java 17 = Major 61)                      |
| **Constant Pool** | 문자열 리터럴, 클래스/메서드/필드 이름, 타입 정보 저장 (가장 큰 영역) |
| **Access Flags**  | public, abstract, final, interface 등 접근 제어 정보         |
| **Class Info**    | 현재 클래스, 부모 클래스, 구현 인터페이스 정보               |
| **Fields**        | 각 필드의 이름, 타입, 접근 제어자                            |
| **Methods**       | 메서드 정보 + 바이트코드 자체가 포함됨                       |
| **Attributes**    | 소스 파일명, 디버깅 정보(라인 번호 매핑), 어노테이션 등      |

> **면접 TIP**: Magic Number가 `CAFEBABE`라는 것과 Constant Pool이 가장 큰 영역이라는 점을 알고 있으면 좋다.



##  클래스 로딩 (Class Loading)

javac이 `.class`를 만든 후, JVM이 바로 실행하는 것이 아니라 **ClassLoader**가 `.class` 파일을 JVM 메모리에 적재하는 과정이 있다.

| 단계               | 설명                                                         |
| ------------------ | ------------------------------------------------------------ |
| **Loading**        | .class 파일을 읽어서 JVM 메모리에 올림                       |
| **Linking**        | Verification(바이트코드 유효성 검증) → Preparation(static 변수 메모리 할당) → Resolution(심볼릭 참조를 실제 참조로 변환) |
| **Initialization** | static 블록 실행, static 변수 초기화                         |





##  바이트코드 실행 — 스택 기반 아키텍처

### 스택 기반(Stack-Based) 실행

JVM은 CPU 레지스터를 직접 사용하지 않고, **오퍼랜드 스택(Operand Stack)** 이라는 자료구조를 사용하여 모든 연산을 수행한다.

### 예시: int c = a + b

```
iconst_3       // 상수 3을 스택에 push       → 스택: [3]
istore_1       // 스택 top을 변수 a에 저장    → 스택: []
iconst_5       // 상수 5를 스택에 push       → 스택: [5]
istore_2       // 스택 top을 변수 b에 저장    → 스택: []
iload_1        // 변수 a 값을 스택에 push     → 스택: [3]
iload_2        // 변수 b 값을 스택에 push     → 스택: [3, 5]
iadd           // 두 값을 pop, 더하고 결과 push → 스택: [8]
istore_3       // 스택 top을 변수 c에 저장    → 스택: []
```

모든 연산이 스택에서 값을 꺼내고(pop), 결과를 다시 넣는(push) 방식으로 동작한다.

### 왜 스택 기반으로 설계했는가?

- **플랫폼 독립적**: CPU마다 레지스터 수와 구조가 다르지만, 스택은 추상적이라 어디서든 동일
- **바이트코드가 단순**: 레지스터 번호를 지정할 필요 없이, 항상 스택 top에서 작업
- **WORA 철학과 일치**

### Stack-Based vs Register-Based 비교

| 항목            | 스택 기반 (JVM)             | 레지스터 기반 (Dalvik)  |
| --------------- | --------------------------- | ----------------------- |
| 연산 방식       | 스택에서 pop/push           | 레지스터 직접 지정      |
| 바이트코드 크기 | 작음 (피연산자 지정 불필요) | 큼 (레지스터 번호 필요) |
| 실행 속도       | 상대적으로 느림             | 상대적으로 빠름         |
| 플랫폼 독립성   | 높음 (WORA)                 | 낮음                    |





##  스택 프레임 (Stack Frame)

메서드가 호출될 때마다 스택 프레임이 생성되며, 다음 세 가지를 포함한다:

### 구성 요소

| 구성 요소                                 | 설명                                                         |
| ----------------------------------------- | ------------------------------------------------------------ |
| **지역 변수 배열 (Local Variable Array)** | this, 매개변수, 지역 변수 저장 ([0]=this, [1]=첫번째 인자...) |
| **오퍼랜드 스택 (Operand Stack)**         | 연산의 중간 결과를 임시 저장하는 작업 공간                   |
| **프레임 데이터 (Frame Data)**            | 리턴 주소, 상수 풀 참조, 예외 테이블                         |

### 스택 프레임 구조 예시

```
JVM Stack
┌──────────────────────┐
│  greet() 스택 프레임    │  ← 현재 실행 중
│  - 지역변수: this, name, message
│  - 오퍼랜드 스택
│  - 프레임 데이터
├──────────────────────┤
│  main() 스택 프레임     │  ← 호출자
│  - 지역변수: args, ...
│  - 오퍼랜드 스택
│  - 프레임 데이터
└──────────────────────┘
```

메서드가 리턴되면 해당 스택 프레임이 pop되고 호출자의 프레임으로 돌아간다. 재귀 호출이 너무 깊으면 스택 프레임이 계속 쌓여 **StackOverflowError**가 발생한다.





##  인터프리터 (Interpreter)

JVM은 처음에 바이트코드를 인터프리터로 실행한다. 인터프리터는 바이트코드를 **한 줄씩 읽어서 해당하는 기계어로 변환하고 실행**한다. 컴파일 과정 없이 즉시 실행할 수 있어 빠른 시작(startup)이 가능하지만, 같은 코드를 반복 실행할 때 매번 해석해야 하므로 성능이 떨어진다.

### 인터프리터의 역할

- **즉시 실행**: 컴파일 대기 없이 바로 바이트코드 실행
- **프로파일링**: 메서드 호출 횟수, 분기 통계 등 런타임 정보 수집
- **Hot Code 감지**: 임계값(약 10,000회)을 넘는 코드를 JIT에게 컴파일 요청





##  JIT 컴파일러

### 핵심 개념

JIT(Just-In-Time) 컴파일러는 인터프리터의 성능 한계를 해결하기 위해, **Hot Code를 네이티브 기계어로 컴파일**한다. 한번 컴파일된 코드는 이후 인터프리터 없이 직접 실행되어 매우 빠르다.

### 처음부터 전부 JIT 컴파일하지 않는 이유

- **컴파일 비용**: JIT 컴파일 자체가 무거운 작업 (최적화 분석, 레지스터 할당 등)
- **시작 시간**: 모든 코드를 한꺼번에 컴파일하면 startup time이 매우 느려짐
- **불필요한 컴파일**: 프로그램에서 실제로 자주 실행되는 코드(Hot Code)는 전체의 일부분

JVM은 **적응적 최적화(Adaptive Optimization)** 로 빠른 시작과 런타임 최적화를 모두 잡는다.





##  Tiered Compilation (계층적 컴파일)

HotSpot JVM은 C1과 C2 두 가지 JIT 컴파일러를 단계적으로 사용한다.

| 항목        | C1 (Client Compiler)                 | C2 (Server Compiler)                           |
| ----------- | ------------------------------------ | ---------------------------------------------- |
| 컴파일 속도 | 빠름                                 | 느림                                           |
| 최적화 수준 | 가벼움 (간단한 인라이닝, 상수 접기)  | 강력함 (루프 언롤링, 탈출 분석, 고급 인라이닝) |
| 목적        | 빠른 시작 + 인터프리터보다 나은 성능 | 최고 수준의 런타임 성능                        |

```
실행 흐름:
Interpreter ──▶ C1 (빠른 컴파일) ──▶ C2 (강력한 최적화)
  (즉시 실행)    (가벼운 최적화)       (최고 성능)
```

빠른 컴파일과 강력한 최적화는 양립하기 어렵기 때문에 역할을 나눈 것이다.



##  JIT 최적화 기법

### 인라이닝 (Inlining) — "최적화의 어머니"

메서드 호출을 없애고 메서드 본문을 호출한 자리에 직접 복사하는 기법이다.

```java
// 인라이닝 전
public int calculate() {
    return add(3, 5) + add(10, 20);
}

// 인라이닝 후
public int calculate() {
    return (3 + 5) + (10 + 20);
}

// 상수 접기(Constant Folding) 추가 적용 후
public int calculate() {
    return 38;
}
```

**효과:**

- **메서드 호출 오버헤드 제거**: 스택 프레임 생성, 매개변수 전달, 점프, 리턴값 처리가 사라짐
- **연쇄 최적화 가능**: 인라이닝 후 상수 접기, 불필요한 코드 제거 등 후속 최적화가 가능해짐

### 기타 주요 최적화 기법

| 기법                                         | 설명                                                     |
| -------------------------------------------- | -------------------------------------------------------- |
| **상수 접기 (Constant Folding)**             | 컴파일 타임에 계산 가능한 식을 미리 계산 (3+5 → 8)       |
| **루프 언롤링 (Loop Unrolling)**             | 반복문의 본문을 복사하여 분기 오버헤드 감소              |
| **탈출 분석 (Escape Analysis)**              | 객체가 메서드 밖으로 탈출하지 않으면 힙 대신 스택에 할당 |
| **분기 예측 (Branch Prediction)**            | 자주 실행되는 분기를 예측하여 파이프라인 효율 향상       |
| **추측적 최적화 (Speculative Optimization)** | 런타임 프로파일링 기반으로 가정을 세우고 최적화          |



##  Deoptimization (역최적화)

JIT의 추측적 최적화가 틀렸을 때 발생한다. JVM은 최적화된 기계어를 버리고 인터프리터 모드로 복귀한 후, 새로운 프로파일링 정보를 기반으로 재컴파일한다.

```
1. JIT: "99% Dog네? → Dog.speak() 인라이닝!"  (최적화된 기계어)
2. 런타임: Cat이 들어옴 → "가정이 틀렸다!"
3. Deoptimization → 인터프리터 모드로 복귀
4. 새로운 프로파일링 → Dog/Cat 모두 반영하여 재컴파일
```

**이것이 가능한 이유**: JVM이 **원본 바이트코드를 항상 보관**하고 있어 언제든 안전하게 돌아갈 수 있기 때문이다. javac이 바이트코드를 원본에 가깝게 유지하는 것이 여기서도 중요해진다.

너무 자주 deoptimization이 발생하는 코드에 대해서는 추측적 최적화를 포기하고 보수적으로 컴파일하기도 한다.



##  전체 실행 흐름 종합

```
1. javac 컴파일
   Hello.java → Hello.class (바이트코드)
   - CAFEBABE, 버전, 상수 풀, 필드, 메서드 등 포함
   - 최소한의 최적화만 수행 (JIT에게 여지를 남김)

2. 클래스 로딩
   ClassLoader가 Hello.class를 JVM 메모리에 적재
   - Loading → Linking (검증/준비/해석) → Initialization

3. 인터프리터 실행
   바이트코드를 한 줄씩 해석하며 실행
   - main()의 스택 프레임 생성
   - 오퍼랜드 스택으로 연산 수행
   - 동시에 프로파일링 정보 수집 (메서드 호출 횟수, 분기 통계 등)

4. JIT 컴파일 (Tiered Compilation)
   Hot Code가 감지되면:
   - C1: 빠르게 컴파일, 가벼운 최적화
   - C2: 느리지만 강력한 최적화 (인라이닝, 탈출 분석 등)
   - 가정이 틀리면 → Deoptimization → 인터프리터로 복귀

5. 출력
   System.out.println("Hello World") 실행 → 화면에 출력
```

### javac, JIT, Bytecode

---

**Q1. Java가 두 단계 컴파일을 하는 이유는?**

> WORA(Write Once, Run Anywhere)를 실현하기 위해 바이트코드라는 중간 표현을 둡니다. javac이 플랫폼 독립적인 바이트코드를 생성하고, 각 플랫폼의 JVM이 해당 환경에 맞는 기계어로 변환합니다. 이식성을 얻는 대신 런타임 변환 비용이 생기며, 이 비용을 줄이기 위해 JIT 컴파일러가 도입되었습니다.

---

**Q2. javac이 공격적으로 최적화하지 않는 이유는?**

> javac은 정적 정보만 가지고 있어 런타임 행동을 모릅니다. 다형성, 실행 빈도, 분기 통계 등의 정보는 JIT만 알 수 있습니다. 또한 바이트코드를 원본에 가깝게 유지해야 JIT가 최적화할 여지가 생깁니다.

---

**Q3. 처음부터 JIT으로 컴파일하지 않는 이유는?**

> JIT 컴파일 자체가 무거운 작업이므로 모든 코드를 한번에 컴파일하면 시작 시간이 매우 느려집니다. 대부분의 코드는 자주 실행되지 않으므로 컴파일이 낭비입니다. 적응적 최적화(Adaptive Optimization)로 인터프리터로 빠르게 시작하고, 임계값(약 10,000회)을 넘는 Hot Code만 JIT 컴파일하여 빠른 시작과 런타임 최적화를 모두 잡습니다.

---

**Q4. C1과 C2 컴파일러의 차이는?**

> C1은 빠른 컴파일과 가벼운 최적화를, C2는 느리지만 강력한 최적화를 담당합니다. Tiered Compilation을 통해 Interpreter → C1 → C2로 단계적으로 최적화 수준을 높입니다. 빠른 컴파일과 강력한 최적화는 양립하기 어렵기 때문에 역할을 나눈 것입니다.

---

**Q5. Deoptimization이란?**

> JIT의 추측적 최적화 가정이 틀렸을 때, 최적화된 기계어를 버리고 인터프리터 모드로 복귀하는 것입니다. 예를 들어 "99%는 Dog"라고 가정하고 인라이닝했는데 Cat이 들어오면 deoptimization이 발생합니다. JVM이 원본 바이트코드를 항상 보관하고 있어 언제든 안전하게 돌아갈 수 있습니다.

---

**Q6. 인라이닝(Inlining)이란?**

> 메서드 호출을 제거하고 메서드 본문을 호출 지점에 직접 삽입하는 기법입니다. 스택 프레임 생성/제거 오버헤드를 없애고, 상수 접기(Constant Folding) 등 후속 최적화를 가능하게 합니다. "최적화의 어머니(mother of all optimizations)"라고 불립니다.

---

**Q7. JVM이 스택 기반인 이유는?**

> CPU마다 레지스터 수와 구조가 다르지만, 스택은 추상적이라 어떤 플랫폼에서든 동일하게 동작합니다. 바이트코드가 단순해지고 WORA 철학과 일치합니다. 반면 Android의 Dalvik VM은 레지스터 기반으로, 실행 속도는 더 빠를 수 있지만 바이트코드가 복잡해집니다.

---

**Q8. 스택 프레임에는 어떤 것이 포함되나요?**

> 지역 변수 배열(this, 매개변수, 지역 변수 저장), 오퍼랜드 스택(연산의 중간 결과 임시 저장), 프레임 데이터(리턴 주소, 상수 풀 참조, 예외 테이블)가 포함됩니다. 메서드가 리턴되면 해당 스택 프레임이 pop되고, 재귀가 너무 깊으면 StackOverflowError가 발생합니다.

---

**Q9. .class 파일에는 무엇이 포함되나요?**

> Magic Number(0xCAFEBABE), 버전 정보, 상수 풀(가장 큰 영역으로 문자열 리터럴, 클래스/메서드/필드 이름 등 저장), 접근 제어 플래그, 클래스/부모 클래스/인터페이스 정보, 필드 정보, 메서드 정보(바이트코드 포함), 어트리뷰트(디버깅 정보, 어노테이션 등)가 포함됩니다.

---

**Q10. 전체 실행 흐름을 처음부터 끝까지 설명해주세요.**

> javac이 .java를 .class(바이트코드)로 컴파일합니다. ClassLoader가 Loading → Linking(검증/준비/해석) → Initialization 순서로 .class를 JVM 메모리에 적재합니다. 인터프리터가 바이트코드를 한 줄씩 해석하며 실행하고, 동시에 프로파일링 정보를 수집합니다. 임계값을 넘는 Hot Code가 감지되면 JIT 컴파일러가 C1(빠른 컴파일) → C2(강력한 최적화) 순서로 기계어로 변환합니다. 가정이 틀리면 Deoptimization으로 인터프리터로 복귀하고 재컴파일합니다.
