## 📝 본문

## 1. Thread-per-Request — 요청 하나당 스레드 하나

Tomcat의 기본 동작 방식은 **Thread-per-Request** 모델입니다. 요청 1개가 들어오면 스레드 1개가 배정되어, 해당 요청의 처리가 끝날 때까지 그 스레드를 점유합니다.

```
요청 1 → [Thread-1] ████████░░░░░░ → 응답    (████ = 처리 중, ░ = DB 기다리는 중)
요청 2 → [Thread-2] ██░░░░░░░░░░██ → 응답
요청 201 → 🚫 Thread 없음! → 대기열(queue)에서 대기
```

은행 창구를 생각하면 이해하기 쉽습니다. 고객 한 명이 오면 창구 직원 한 명이 전담하고, 그 고객의 일이 끝날 때까지 다른 고객은 받을 수 없습니다. DB 호출로 3초를 기다리는 동안에도 스레드가 반환되지 않고 점유된 상태로 유지됩니다.

------



## 2. Thread Pool — 스레드의 재사용

스레드를 요청마다 새로 만들면 생성/소멸 비용이 큽니다. 그래서 미리 스레드를 N개 만들어놓고, 요청이 오면 빌려주고, 끝나면 반납받는 **Thread Pool** 방식을 사용합니다.

```yaml
# application.yml
server:
  tomcat:
    threads:
      max: 200          # 최대 스레드 수 (기본값 200)
      min-spare: 10     # 항상 대기 중인 최소 스레드 수
    max-connections: 8192  # 최대 동시 연결 수
    accept-count: 100      # 스레드가 모두 사용 중일 때 대기열 크기
```

여기서 `max-connections`와 `max-threads`의 차이가 중요합니다. 연결(connection)은 8192개까지 맺을 수 있지만, 동시에 **실제로 일하는** 스레드는 200개뿐입니다. 나머지 요청은 큐에서 차례를 기다립니다.

요청 흐름을 정리하면 다음과 같습니다.

```
요청 도착 → 연결 수 < 8192? → Thread Pool에 여유 스레드 있음? → 즉시 처리
                                    ↓ 없음
                              accept-count(100) 대기열에 추가
                                    ↓ 대기열도 가득 참
                              Connection Refused (요청 거부)
```

## 3. Blocking I/O — 스레드가 멈추고 기다리는 방식

Blocking I/O란, 외부 작업(DB 조회, API 호출, 파일 읽기)의 결과가 올 때까지 현재 스레드가 아무것도 하지 않고 기다리는 방식입니다.

```java
// 이 한 줄이 실행되는 동안 현재 스레드는 완전히 멈춤
User user = jdbcTemplate.queryForObject("SELECT * FROM users WHERE id = ?", ...);
// DB가 응답할 때까지 이 스레드는 아무것도 하지 않고 대기 (보통 1~100ms)
```

전화 주문에 비유하면, "잠시만요 확인해볼게요~" 하고 보류 상태가 된 것과 같습니다. 상대방이 다시 말할 때까지 전화기를 붙잡고 아무것도 할 수 없습니다.

### 왜 문제인가?

200개 스레드가 전부 DB 응답을 기다리고 있으면, 201번째 요청부터는 처리할 스레드가 없어 대기열에 쌓입니다. I/O 대기 시간이 길어질수록(느린 DB 쿼리, 외부 API 타임아웃 등) 이 문제가 심각해집니다.

### Non-blocking I/O — 반대 개념

Non-blocking I/O는 요청만 던져놓고 스레드가 다른 일을 하러 갑니다. 결과가 준비되면 콜백이나 이벤트로 알려줍니다. 이것이 Spring WebFlux가 채택한 방식이며, 별도 글에서 자세히 다룹니다.

## 핵심 키워드

| 키워드                        | 설명                                                |
| ----------------------------- | --------------------------------------------------- |
| **Thread-per-Request**        | 요청 1개당 스레드 1개를 배정하는 처리 모델          |
| **Thread Pool**               | 미리 만들어둔 스레드들의 재사용 가능한 묶음         |
| **server.tomcat.threads.max** | Tomcat 최대 스레드 수 설정 (기본값 200)             |
| **max-connections**           | 동시에 맺을 수 있는 최대 연결 수 (기본값 8192)      |
| **accept-count**              | 스레드가 모두 사용 중일 때 대기열 크기 (기본값 100) |
| **Blocking I/O**              | 외부 작업 결과를 기다리는 동안 스레드가 멈추는 방식 |
| **Non-blocking I/O**          | 요청을 던져놓고 스레드가 다른 일을 하는 방식        |



## 🎯 면접 예상 질문

### Q1. Tomcat의 Thread-per-Request 모델을 설명하고, 이 모델의 한계는 무엇인가요?

**모범 답안:**

> Thread-per-Request는 요청 하나당 스레드 하나를 배정하는 모델입니다. 이 모델의 한계는 스레드 수가 유한하다는 점입니다. Tomcat 기본 스레드가 200개인데, 200개가 모두 사용 중이면 이후 요청은 대기열에서 기다려야 합니다. 여기에 Blocking I/O가 결합되면, DB 응답을 기다리는 동안에도 스레드가 반환되지 않아 스레드 고갈이 더 빨리 발생합니다.

### Q2. Thread Pool이란 무엇이며, 왜 스레드를 매번 생성하지 않고 풀로 관리하나요?

**모범 답안:**

> Thread Pool은 미리 스레드를 만들어두고 재사용하는 방식입니다. 스레드 생성 시 OS 커널에 스레드를 등록하고, 스택 메모리(기본 약 512KB~1MB)를 할당하는 비용이 발생합니다. 요청마다 생성/소멸을 반복하면 이 오버헤드가 누적되어 성능이 저하됩니다. Thread Pool은 미리 스레드를 만들어두고 재사용함으로써 이 비용을 제거합니다.

### Q3. Blocking I/O가 동시 처리 성능에 미치는 영향을 설명해주세요.

**모범 답안:**

> Blocking I/O 환경에서는 DB 조회나 외부 API 호출 시 해당 스레드가 응답을 받을 때까지 멈춘 상태로 유지됩니다. 각 스레드는 독립적으로 동작하므로 한 스레드의 Blocking이 다른 스레드에 직접 영향을 주지는 않습니다. 하지만 동시에 많은 요청이 I/O 대기 상태에 빠지면 Thread Pool이 고갈되어, 이후 들어오는 요청은 처리할 스레드가 없어 대기하게 됩니다. 이것이 동시 처리 성능의 병목이 됩니다.

**주의할 점:**

- "다수의 스레드가 하나의 Servlet을 실행할 때 Blocking이 발생하면 다른 스레드도 대기한다"는 **틀린 설명**입니다. 각 스레드는 독립적으로 동작하며, 한 스레드의 Blocking은 다른 스레드에 영향을 주지 않습니다.
- Blocking I/O의 진짜 문제는 스레드 간 간섭이 아니라 **Thread Pool 고갈**입니다.
